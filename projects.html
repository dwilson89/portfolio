Project: Silo Inspection UAV
Year: 2014
Role: Ground Control Station and Localisation Developer
Languages Utilised: C++, Arduino
Project Description: Develop a UAV system to successfully navigate in an indoor environment, and locate potential pests (Gypsie Moths, Straw).
Code Desciption:

For this project, I worked on two different systems that interacted with one another and several other subsystems, a Ground Control Station application and a Localisation system which could localise an UAV in an indoor environment.

Using C++ and the Arduino programming language I developed a application that could interface with a Vicon (Motion capture) system, and the Autopilot and Telecommunications, and the Ground Control Station in order to localise itself to the area in which it was flying. The application was developed to work with a series of Arduinos, one on the ground and one attached to the UAV itself. My application was designed and developed to be able to connect to a vicon server and an attached serial device (Arduino) and then periodically query the server for current locational data being generated by the UAV. From here the data is converted to Mavlink messages (a form of telecommunication protocols for UAVs), and via the use of a serial connection the application sends these messages to the arduino on the ground which has an serial radio plugged into it. From this point the messages are then broadcasted to another radio attached to the UAV, which is plugged into the other arduino. This arduino is programmed to act like an GPS system and communicate with the autopilot subsystem through its GPS port. It is designed to listen to the radio for that particular message being broadcast by ground control, and once detected process it and turn it into "fake" gps messages using UBX protocol. These messages are then sent to the autopilots GPS port. Videos and imagery of this project can be found below.

Project: Airport Simulator
Year: 2014
Role: Developer
Languages Utilised: C
Project Description: Build and design a program to simulate a one runway airport, that has planes arriving and departing on random intervals.
Code Description:

The application is designed to have three different processes running at all times. A "producer" (process to create arriving airplanes), a "consumer" (process to destroy/depart airplanes), and a "observer" to allow the user to view current airport information (current capcity and airplane information) and/or exit the application. In addition to this thier was a limit to airport capacity (10 at any time), and one runway which could only produce/consume one at a time. I was in charge of development of functionality to implement the "producer" in order to create and simulate incoming aircraft. Airplanes are randomly created/destroyed via the use of initial percentage values (inputted by the user at runtime) to determine the chances of an incoming/departing flight. If an airplane is to be created it checks the current capacity of the airport and then waits for the runway to be clear. From here a "new" airplane is created with a code that is randomly generated and then put into an empty flight bay randomly determined.

In addition to this I was also in charge of implementation of protective measures, such as semaphores and mutexes in order to protect shared data and remove the risk of currupting it. They were also used as a mean to block the production/consumation of airplanes once the capacity had reached its limits.

Project: Hadamard Decoding/Encoding Application
Year: 2013
Role: Developer
Languages Utilised: C
Project Description: Design and implement a series of functions that encrypt/decrypt files, via the use of a Hadamard matrix.
Code Description:

This application has several different options built into it, which is chosen by the user at run time, and is designed to primarily work with PGM and txt files. As it is a console application, it requires the user to input some initial data: hadamard [option] {format u v depth}, format refering to PGM, u and v image size, and depth max pixel intensity. The options include:

  • e which will encrypt the file 1 into file 2 using u and v.
  • d decrypt file 1 and place in file 2 using format u, v and depth
  • p pack encoded data from file 1 into file 2
  • u unpack encoded data from file 1 into file 2
  • m extract alien message from file 1 and place in file 2 using u and v

The file encryption and decryption is designed and implemented to work by using a hadamard matrix and replacing each bytes value with a corresponding entry in that matrix. In order to detect corruption the values are compared and if the ith value is found to be the best match, the bytes value is i.

The packing options cuts the size to 7/8th of the original size via the use of bit shifts and by removing padded zeroes. This is then reversed if the unpacking option has been chosen, by using an reverse method of bit shifts and padding.

If option m is chosen, a special 32 character long "alien" aphabet is used, in conjunction with a different encryption process using a 64bit value. This process involves detecting and finding corrupted bits in the value, and splitting the 64 bit value into two 32 bit halfs, starting from the left (for both sections) the corrupted bits ith location corresponds to the alphabet lettering positions (eg. ith = 4, would be H). This is to be used with .txt files.

Project: Teensy Pong and Breakout
Year: 2013
Role: Developer
Languages Utilised: C
Project Description: Design and implement, the games pong and breakout for the teensy micoprocessor board.
Code Description:

This application was written to work on a teensy microprocessor board, a LCD screen, 3 buttons and a potentiometer. These games classic Pong and Breakout, were designed and implemented to behave and work exactly like the original games, however with few changes inorder to work on the teensy microprocessor board. These game modes were achieved via the use of micro second delays and constant game state checking, and LCD updating functions. The potentiometer was designed as a way that when adjusted it increases the speed/difficulty of the game. Additional functionality was added to included the option to set the LCD backlighting (and saved in EEPROM), additional lives for breakout, and a scoring system for both games.

Project: Travelling Salesman Problem Application
Year: 2013
Role: Developer
Languages Utilised: C++
Description: Design and implement two efficient different methods of calculating the shortest path to be taken by a traveling salesman.
Main tasks:

This application calculates the shortest path to be taken by a traveling salesman, with three different approaches. Brute Force which can only handle 10-12 cities, it computes recursively every combination of tour, finding the minimum.

Memoisation which can handle up to 20-23 cities, runs alot faster but does so at the cost of space as it requires a N x(1 << N) matrix, this is to remove the need to check paths that are similar and already exist.

The approximation can handle very large number of cities, and relies on two well-known graph algorithms; The Minimum Spanning Tree and the Depth First Traversal, in order to provide an approximation to the problem.

Project: Poker Hand Evaluator
Year: 2013
Role: Developer
Languages Utilised: C++
Project Description: Design and implement an application to determine the best Poker hand amongst n number of players.
Code Description:

This console application is designed to determine who has the best hand in poker out of n players, using standard rules for hand rankings. The application can take input from a text file or can randomly create n number of hands, via creating a new randomly shuffled deck and then dealing them. The program determines hand types from evaluating each players hand. For each type of hand a different algorithm has been established in order to determined which player has the best hand of that type. Once hand types and values have been determined, the application then ranks them in the correct order, determined by the rules of the game.
